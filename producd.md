# 🧠 “大脑外脑” (Brain-Sync) - 个人知识管理 App PRD v1.0

## 一、 产品概述 (Product Overview)

### 1.1 产品背景与痛点
*   **信息过载**：每天面对大量的科技博客（如 OpenAI Blog）和播客更新，没有时间全部阅读/收听。
*   **收藏吃灰**：看到好文章习惯性收藏，但缺乏提炼，最终沦为“信息垃圾场”。
*   **知识不成体系**：工作、投资、个人爱好等知识散落在不同软件中，难以形成系统的“第二大脑”。

### 1.2 产品愿景
打造一个**“先过滤、再沉淀”**的闭环系统。利用 Qwen 大模型作为“数字助理”，每天替你初步消化 RSS/播客 信息，只有经过 AI 提炼且你认为有价值的内容，才会被结构化地沉淀到个人知识库中。

### 1.3 目标用户与使用场景
*   **用户**：仅限个人开发者/创作者本人（单用户系统）。
*   **移动端场景（碎片时间）**：早晨通勤时，像刷朋友圈一样刷 AI 总结好的播客和博客，快速筛选并一键入库。
*   **桌面端场景（深度时间）**：在电脑前打开网页，对入库的知识进行深度编辑、归类和二次输出。

---

## 二、 核心业务流程 (User Flow)

1.  **信息抓取**：服务器后台每天定时拉取配置好的 RSS 源（文章摘要或播客 Show Notes）。
2.  **AI 预处理 (按需触发)**：用户在前端点击某条信息，触发 Qwen 大模型进行“翻译 + 总结 + 见解生成”。
3.  **分发与归档**：用户阅读 AI 总结后，判断价值：
    *   *无价值*：直接划走/忽略。
    *   *有价值*：点击“一键入库”，选择对应的知识库分类（如：AI技术）。
4.  **知识内化**：进入知识库模块，查阅已归档的内容，或手动新建个人笔记。

## 三、 功能需求明细 (Functional Requirements)

产品主要分为三大模块：**【信息流 (Feed)】**、**【知识库 (Vault)】**、**【系统设置 (Settings)】**。

### 模块一：每日信息流 (Feed) —— “数字助理的汇报”

这是你每天打开 App 最先看到的页面。

| 功能名称 | 功能描述 | 交互与规则说明 |
| :--- | :--- | :--- |
| **RSS 列表展示** | 以信息流形式展示当日或未读的 RSS 抓取内容。 | 列表显示：文章/播客标题、来源网站、发布时间。按时间倒序排列。 |
| **AI 深度解析** | **(核心功能)** 点击单条信息后，调用 Qwen 模型对文章正文或播客简介进行处理。 | **AI 输出固定格式要求：**<br>1. **【标题翻译】**：若原标题为英文，提供精准中文翻译。<br>2. **【核心总结】**：用 3 个带序号的要点提炼核心内容，拒绝冗长。<br>3. **【专属见解】**：结合用户的4大知识领域，给出一句点评（例：“此项 AI 技术可用于提升你的工作效率”）。 |
| **一键入库** | 将当前 AI 处理好的内容保存至个人知识库。 | 底部提供“保存至知识库”按钮。点击后弹出 4 大主分类供选择。保存内容包括：原文链接、原标题、AI 总结的全部文本。 |
| **状态标记** | 区分已读/未读/已入库的内容。 | 已入库的内容变灰或打上“已归档”标签，保持信息流清爽。 |

### 模块二：个人知识库 (Vault) —— “结构的永久记忆”

这里是你个人的数字资产保险箱。

| 功能名称 | 功能描述 | 交互与规则说明 |
| :--- | :--- | :--- |
| **四大核心分类** | 系统硬编码 4 个一级目录，强制分类，避免层级过深导致混乱。 | 1. **💼 工作能力** (工作复盘、专业技能)<br>2. **🤖 AI 技术** (大模型动态、提示词、工具)<br>3. **📈 投资** (宏观经济、理财策略)<br>4. **🌟 个人提升** (下设标签：摄影、运动、读书计划) |
| **Markdown 笔记** | 支持创建纯文本/Markdown 格式的笔记。 | 无论是从 Feed 归档过来的，还是手动创建的，全部使用 Markdown 渲染，支持粗体、列表、代码块等基础排版。 |
| **标签系统 (Tags)** | 为笔记添加更细颗粒度的标签。 | 一篇笔记只能属于 1 个核心分类，但可以有多个 Tag（如 `#2024读书`, `#OpenAI`）。 |
| **全局搜索** | 快速查找所需知识。 | 提供顶部搜索框，支持对笔记“标题”和“正文内容”的关键词模糊匹配搜索。 |

### 模块三：系统设置 (Settings)

| 功能名称 | 功能描述 | 交互与规则说明 |
| :--- | :--- | :--- |
| **RSS 源管理** | 添加、删除、编辑订阅源。 | 输入 RSS 链接即可添加（支持博客和播客源）。 |

## 四、 非功能需求与技术约束 (Non-Functional Requirements)

针对你**“小型服务器、资源少、无域名、多端跨平台”**的现状，制定以下策略：

### 4.1 极简跨平台方案 (无须应用商店)
*   **前端采用 PWA (Progressive Web App)**：只写一套基于移动端 UI 适配的 Web 网页代码。
*   **使用方式**：手机浏览器（Safari/Chrome）打开网页 -> 点击“添加到主屏幕” -> 生成独立的 App 图标。打开后隐藏浏览器地址栏，沉浸感等同于原生 App。电脑端直接通过浏览器访问。
*   **免域名托管**：前端代码一键免费部署到 **Vercel**（自带稳定的国外二级域名，国内可直连，解决前端无域名问题）。

### 4.2 超轻量级后端方案 (保护小服务器)
*   **服务器只跑后端接口 (API)**：使用 `Python (FastAPI)` 编写。Python 处理 AI 请求和 RSS 抓取效率极高，内存占用小。
*   **放弃重型数据库**：不安装 MySQL。直接使用 **`SQLite`**。它是一个单文件数据库，对个人几十万字的知识库来说，性能完全溢出，且直接下载那个 `.db` 文件就能完成整站数据备份。
*   **播客处理策略**：为了不撑爆小服务器带宽和内存，**不下载播客音频文件**。系统只抓取播客 RSS 中的 `Show Notes (节目文字简介)` 交给 AI 处理，这足以提炼核心信息。

### 4.3 安全性 (Security)
*   因为是一个人用的系统，不需要复杂的注册/找回密码逻辑。
*   全站增加一个全局的**“访问密码 (Access Token)”**。打开网页时输入一次正确密码，缓存在本地，之后即可顺畅使用。防止陌生人访问你的服务器消耗你的大模型 API 额度。
